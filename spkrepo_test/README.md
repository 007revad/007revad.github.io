# spkrepo

**DSM Package Center repository** for 007revad's Synology packages.

Add to DSM Package Center → Settings → Package Sources:

| Name | Location |
|------|----------|
| `007revad` | `https://spkrepo.007daver.workers.dev/` |

---

## How it works

DSM Package Center sends a GET request to the Location URL with query parameters
(`arch`, `major`, `minor`, `build`, etc.). A Cloudflare worker filters the GitHub index.json 
file by `arch` and `firmware` to create a list of packages that match the Package Center 
request and replies to Package Center.

The Cloudflare https://spkrepo.007daver.workers.dev/ worker only sends DSM packages from 
GitHub `index.json` that match the requested arch and DSM version.

The `index.json` is auto-regenerated by the
[Update spkrepo index.json](../../.github/workflows/update-spkrepo.yml) workflow
whenever any of the monitored GitHub repos with packages publish a new release.

The GitHub workflow streams the first 512KB of the spk files to get the INFO file and 
PACKAGE_ICON_256.PNG. It then creates a 120x120 pixel <package>_120.png for Package Center 
and adds the package's information from the INFO file to index.json.

The Transcode package ships two separate SPKs per release (`-DSM-7.2-` and `-DSM-7.3-`), each
with a `firmware` / `os_max_ver` range so DSM only offers the matching build. The workflow
matches assets by the `DSM-<ver>-geminilakenk.spk` pattern rather than a fixed filename.

## Adding a new package

1. Add a new `make_entry` call in `.github/workflows/update-spkrepo.yml`
2. Add the `notify-spkrepo` job to that package repo's release workflow
3. Run the `update-spkrepo.yml` workflow manually once to regenerate `index.json`
